# 影响力叙事工具箱

## 1. 基础改写公式

### STAR / CAR

```text
为了[业务目标或技术挑战]，
我[采取了什么关键动作]，
最终带来[可量化或可感知的结果]。
```

示例：

- 修改前：负责订单系统开发
- 修改后：为解决大促期间订单查询缓慢问题，主导设计并落地基于 Redis 的缓存方案，将平均响应时间从 800ms 降至 120ms，并支撑峰值 10 万 QPS。

## 2. 决策-权衡写法

当经历里存在技术选型、架构权衡、复杂约束时，优先用这个结构：

```text
为解决[问题]，
评估了[方案 A] 与 [方案 B]。
我选择[方案]，因为[关键理由]，
并通过[补偿措施或实现细节]控制风险，
最终实现[结果]。
```

示例：

- 修改前：使用 Kafka 实现消息队列
- 修改后：为解决跨服务数据一致性问题，在 RabbitMQ 和 Kafka 之间进行技术选型。考虑到未来吞吐增长和实时计算需求，选择 Kafka，并设计幂等消费机制降低重复消费风险，最终支撑日均 5000 万条消息处理且未出现数据丢失。

## 3. 常见改写方向

### 从职责改成成果

- 修改前：负责用户模块开发和维护
- 修改后：主导用户模块重构，拆分服务边界并补齐单测，使新功能交付周期从 2 周缩短到 3 天，单测覆盖率从 30% 提升到 85%。

### 从"做事"改成交付产物

- 修改前：负责搭建内部工具
- 修改后：设计并交付内部发布平台，将原本分散在多个脚本中的发布流程统一到一个可复用平台中，覆盖研发与测试团队的日常发布操作。

### 从过程改成价值

- 修改前：优化页面性能
- 修改后：通过代码分割、懒加载和 CDN 优化，将首屏加载时间从 4.2 秒降至 1.1 秒，跳出率下降 15%。

### 从"做了什么"改成"为什么这样做"

- 修改前：开发数据处理 pipeline
- 修改后：设计并实现基于 Spark 的实时数据处理 pipeline，日均处理 10TB 用户行为数据，将数据延迟从 T+1 缩短至分钟级，为运营侧提供实时决策支持。

## 4. 没有量化数据时怎么写

如果用户拿不出精确数字，不要硬编。可以换成这些结果表达：

- 建立了可复用标准
- 消除了某类线上风险
- 让某项流程从手工变成自动化
- 让排障、观测、发布更可控
- 让跨团队协作更顺畅

示例占位符：

```text
[量化数据待补：例如每周节省 6 小时人工处理时间]
```

## 5. 量化优先级

优先从这些角度找数字或可验证变化：

1. 性能：延迟、吞吐、成功率、可用性、错误率
2. 业务：转化率、GMV、留存、订单量、收入、客诉量
3. 效率：人天、发布时长、排障时间、交付周期、手工步骤
4. 规模：用户数、接口量、数据量、服务数、团队覆盖范围
5. 风险：事故次数、故障恢复时间、重复问题、误操作概率

如果拿不到精准数字，降级为：

- 高频/低频
- 分钟级/小时级/天级
- 单团队/多团队/全公司
- 核心链路/关键系统/基础设施

## 6. 产物导向模板

当原文只有职责，没有明确交付物时，优先套下面结构：

```text
为了解决[问题]，
我设计/重构/落地了[产物]，
被[团队/系统/用户]使用，
最终带来[结果]。
```

常见产物包括：

- 系统
- 平台
- 服务
- 工具
- 组件
- 规范
- 机制
- 流程
- 看板
- 自动化脚本

## 7. 启发式提问

当原始材料太弱时，用问题把亮点挖出来：

- 这个项目里最难的问题是什么
- 为什么当时选这个方案，而不是另一个方案
- 你真正交付给团队或业务的产物是什么
- 你的改动到底让谁受益了
- 没有你的这项工作，会造成什么问题
- 有没有一个可以证明结果的数字、比例、范围或变化

## 8. 软技能的硬证据化

当经历涉及跨团队协作、辅导新人、项目管理等软技能时，避免空泛表述，改用可验证的行为证据。

### 从空泛改成行为证据

- 修改前：具备良好的跨团队沟通能力
- 修改后：推动前端、后端和数据团队三方对齐接口规范，通过编写 RFC 和主持 4 次评审会，使跨团队联调时间从 2 周缩短到 3 天。

- 修改前：有 mentoring 经验，辅导过初级工程师
- 修改后：为团队 3 名应届生制定 6 个月 onboarding 计划，通过代码评审和每周 1:1 辅导，使其独立交付首个功能模块的平均时间从 3 个月缩短到 6 周。

- 修改前：有一定项目管理能力
- 修改后：主导 XX 项目的整体交付节奏，协调 5 人跨职能团队，通过周迭代规划和风险评估，在 4 个月内完成从需求到上线的全部里程碑。

### 软技能改写核心公式

```text
[具体行为] + [影响范围/涉及方] + [可感知的结果变化]
```

常见软技能及对应证据锚点：

| 软技能 | 证据锚点 |
|--------|----------|
| 沟通能力 | 推动 X 方对齐、主持评审、编写规范/RFC |
| 领导力 | 主导项目交付、协调团队、推动技术决策 |
| Mentoring | 辅导人数、onboarding 计划、缩短成长周期 |
| 项目管理 | 交付时间线、里程碑管理、风险识别与应对 |
| 跨团队协作 | 涉及团队数、对齐的接口/流程、联调效率变化 |

### 判断标准

一条好的软技能表述必须满足：

- 能否想象出具体的行为画面
- 是否涉及其他人或团队
- 是否有可比较的前后变化

一条好的项目描述，至少满足下面三项中的两项：

- 说清了问题
- 说清了你的动作
- 说清了结果
