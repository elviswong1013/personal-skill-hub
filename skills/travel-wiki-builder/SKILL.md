# Travel Wiki Builder

将"搜索资料 → llm-wiki 知识库构建 → 多格式文稿输出"封装为可复用技能。适用于任意旅行目的地，根据目的地类型自动适配行程框架。

## When to Invoke

- 用户要求为某个目的地做完整的旅行攻略（含行程、住宿、交通、预算）
- 用户要求生成旅行计划 PPT、Markdown 或 HTML 文档
- 用户希望构建结构化的旅行知识库（llm-wiki 六层结构）
- 用户提到"按 llm-wiki 方式做旅行攻略"

## When NOT to Invoke

- 简单的单一问题查询（"冰岛天气怎么样？""东京有什么好吃的？"）—— 直接回答即可
- 用户只想看某个景点介绍，不需要完整攻略
- 用户已有现成行程，仅需格式化排版
- 纯闲聊式旅行话题，不涉及文档输出

## 核心流程

```
搜索资料 → raw 层 → source 层 → entity 层 → concept 层 → wiki 层 → PPT/MD/HTML 输出
  (4-6次)  (每主题1文件)  (跨来源提炼)  (实体卡片)  (概念笔记)  (详细攻略)  (3格式)
```

---

## Phase 1: 搜索与资料收集

针对用户指定的旅行目的地，执行 **4-6 次 Web 搜索**，覆盖以下主题：

| # | 搜索主题 | 搜索方向 |
|---|---------|---------|
| 1 | 天气与季节 | 当地季节气候、穿衣指南 |
| 2 | 交通出行 | 租车/自驾/包车/公共交通，驾照与保险要求 |
| 3 | 酒店住宿 | 区域推荐、各档酒店、价格参考 |
| 4 | 景点推荐 | 必去景点、行程路线 |
| 5 | 餐厅美食 | 本地美食、餐厅推荐 |
| 6 | 预算汇率 | 旅行花费、实时汇率 |

- 中英文关键词并行搜索
- 优先使用官方或高权威来源
- 搜索结果保留 URL 来源以供 raw 层标注

## Phase 2: 构建 llm-wiki 六层结构

### 目录结构

在目标目录下创建六层文件夹及根导航文件：

```text
/raw        ← 原始资料（含来源追溯）
/source     ← 综合资料（跨来源提炼）
/entity     ← 实体卡片（酒店/景点/餐厅等）
/concept    ← 概念笔记（核心旅行概念）
/wiki       ← 详细攻略（用户直接阅读）
/history    ← 治理记录
index.md    ← 根导航入口
```

初始化时创建所有目录（跨平台通用方式：使用项目可用的文件系统工具创建文件夹）。

### 2.1 Raw 层（原始资料）

为每个搜索主题创建一个 raw 文件，命名规范：`[目的地]-[主题]-raw.md`

每份 raw 笔记必须包含：
- `## Source Metadata`（来源 URL、收集日期、语言）
- `## Key Facts`（核心数据要点）
- `## First-Pass Judgment`（价值评估 + 候选提升目标 `[[wikilink]]`）

### 2.2 Source 层（综合资料）

跨来源提炼，文件覆盖：目的地概览、交通方式对比、住宿策略、景点指南、餐饮指南。

每份 source 文件需引用 `[[raw/...]]` 原始资料，并包含导航链接。

### 2.3 Entity 层（实体卡片）

记录可持久化的实体：酒店列表（按城市和档次分层）、租车公司、景点（按区域分组）、餐厅（按类型分类）。

### 2.4 Concept 层（概念笔记）

记录稳定的旅行概念，根据目的地类型选择相关概念。每个概念笔记格式：
**定义** → **为什么重要** → **关键信息** → **常见混淆** → **关联概念**

### 2.5 Wiki 层（详细攻略）

**这是用户最直接阅读的层，必须尽量详细。** 文件覆盖：行程方案（至少 3 套：快速/经典/深度）、酒店详解（每家 3-5 行说明含预订建议）、交通指南（自驾和包车全覆盖）、景点推荐（每个景点详细描述含评分/费用/停留时间/安全提示）、餐厅推荐（含超市自炊指南）、打包清单（分类详解）、预算规划。

预算规划要求：基于实时搜索的汇率，覆盖 2-4 人出行场景，至少包含 3 种货币，分经济/舒适/豪华三档。

### 2.6 根导航 index.md

包含快速导航（按层分类链接）和推荐阅读路线。

### 2.7 History 层（治理）

包含项目章程、操作手册和变更日志。

## Phase 3: 多格式文稿输出

基于 wiki 层内容，生成三种格式的演示文稿。具体样式规范参见 `references/color-scheme.md`。

### 3.1 Markdown 文稿

生成约 19 个 section 的 .md 文件，涵盖封面 → 季节优势 → 行程总览 → 每日详情 → 住宿/交通/预算/打包 → 安全须知 → 结束页。

### 3.2 HTML 文稿

深色主题的独立 .html 文件，全屏 slide 布局，CSS Grid 卡片，响应式设计。样式规范见 `references/color-scheme.md` 的 HTML 规格部分。

### 3.3 PPT 文稿

生成 16:9 宽屏 .pptx 文件。使用项目可用的 PPT 生成库，遵循 `references/color-scheme.md` 中定义的配色方案和布局规格。封面页到结束页共约 19 页。

## Phase 4: 目的地类型适配

根据用户的目的地，自动选择行程框架。**在 Phase 2 开始前判断**：

| 目的地类型 | 判断条件 | 行程框架 | 概念层调整 |
|-----------|---------|---------|-----------|
| 环岛/环线型 | 冰岛、台湾、新西兰南岛等 | 7-10 天环线，每日一站 | 启用"环线公路"概念 |
| 单城深度型 | 巴黎、东京、伦敦等 | 3-5 天区域探索 | 改为"区域探索"概念，不使用环线框架 |
| 多城串联型 | 意大利、日本关西等 | 每城 2-4 天，城市间交通串联 | 启用"城市串联"概念，交通工具相应调整 |

不确定时，向用户确认目的地类型和行程偏好。

## Phase 5: 完整性检查

输出完成后验证：

### 文件清单验证
确认所有层级的文件均存在，且输出文件（.md + .html + .pptx）已生成。

### 质量验收
- [ ] 所有 wikilink `[[...]]` 格式正确
- [ ] wiki 层文件内容足够详细
- [ ] 预算包含实时汇率 + 多币种 + 多人数
- [ ] 安全警告用醒目格式标注
- [ ] HTML 在浏览器中可正常渲染
- [ ] PPT 成功生成

---

## 安全内容要求

以下内容必须在 wiki 和文稿中醒目标注：
- 危险自然现象（如冰岛 Sneaker Waves）
- 极端天气警告
- 法律禁区（如 off-road 禁令）
- 保险免责条款

---

## 注意事项

### 汇率处理
每次执行时必须通过搜索获取最新汇率，提供多币种对照表。

### 语言
- wiki 知识和文稿默认中文
- 地名、餐厅名保留原文
- 货币缩写保留国际标准

## Verification Scenarios

### Scenario A: 环岛型目的地

**输入**: "做冰岛夏季 8 天旅行攻略"
**预期**: 启用环线框架，7-10 天行程，concept 层包含环线公路概念，wiki 层生成环岛行程。

### Scenario B: 不触发场景

**输入**: "冰岛天气怎么样？"
**预期**: 不触发本 skill — 属于简单查询，直接回答即可。

### Scenario C: 单城型目的地

**输入**: "做东京 5 天旅行攻略"
**预期**: 启用单城区域探索框架，不使用环线结构，行程为区域探索型。
